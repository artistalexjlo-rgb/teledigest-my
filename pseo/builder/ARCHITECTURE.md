# Карта: мозг ключей → потребители

Единственный источник этой карты. **Меняешь код ключей/потребителей — правишь ЭТУ таблицу в том же коммите.** Чтобы не грепать заново каждый раз.

## Термины

- **сиська** = `keybroker.py` — держит молоко: ключи, квоты (`usage`), часы (`key_clock`), per-key cooldown (бан→разбан). Ключи живут ТОЛЬКО здесь.
- **сосок** = `keybroker.call(user, sysprompt, consumer, model, role)` — **одна** дверь к Gemini: `acquire → HTTP → report → разбор JSON`. Молоко течёт через неё всем. Рот получает `dict`, ключа не видит.
- **рот** = call-site потребителя, где он присасывается к соску. Рот называет себя (`consumer`) и молоко (`model`).
- **драйвер** = кто спавнит потребителя процессом (`runner`/`lang_runner`/`batch`). Спавн обнуляет in-memory такт → темп держит только персистентный `key_clock` в сиське.

## Живые рты (есть драйвер) — 7. Экстрактор НЕ здесь (см. ниже)

Все на `gemini-3.1-flash-lite`, роль `background`. У КАЖДОГО рта — персональный кап (потолок RPD/день). Кап — не оптимизация, а **предохранитель**: сломанный рот (цикл/шторм) не съест больше своей доли, не осушит пул, не заморит остальных. Мозг видит `consumer` → сверяет дневной счёт с капом → сверх капа отказ. Незаписанный рот → `DEFAULT_CONSUMER_CAP` (никогда не uncapped).

**Метод капа (не с потолка):** `кап = дневной_пик_рта × запас(≈3)` — накрыть легальные всплески (новый канал, догон экстрактора, дисперсия 191–482 мухи/день), но остаться сильно ниже пула (12×440 = **5280**). Бэкфилл (×30–50) — НЕ дефолт, а явно поднятый кап на время прогона.

### Бэкфилл ≠ daily (основа капов)

Все рты едят **муху** (`ai_lesson` ≈ 354 симв ≈ **88 ток**, замер 2026-07-16, 14 288 шт).
- **Дневной приток мух ≈ 300/день** (замер: 191–482). При 1 мухе/запрос это ~300 facet-запросов/день = **~25 на ключ из 500 RPD → крохи.** В steady-state ВСЕ рты тривиальны.
- **Большие объёмы (facet 14 288, translate ×языки) — РАЗОВЫЙ БЭКФИЛЛ**, не дневная нагрузка. Потому дефолтный кап мал (дневной пик ×3), а бэкфилл — явный временный подъём.
- Оптимум батчей поэтому нужен **только ради скорости бэкфилла**, не для дневной работы.

## Очередь пула — держит МОЗГ (канон юзера 2026-07-20)

Раннеры снесены как класс (PROCESSES.md) — прежний пункт «очередь держит раннер» мёртв.
**Очередь держит сам мозг: одна выдача в полёте на весь пул.** Рты подходят к соску ПО
ОДНОМУ, кем бы ни были запущены: `acquire` не выдаёт ключ, пока предыдущий грант не
закрыт `report`'ом (бронь `busy_*` в `broker_global`; протухает за `BUSY_TIMEOUT=90с`,
если процесс сдох между grant и report).

- Суммарный темп с IP = естественная латентность HTTP (~15–25/мин) — по-IP 429-волны
  (инциденты 07-19/20: параллельные рты давали 32–45/мин) исчезают конструкцией.
- Порядок обслуживания — по факту прихода (арбитраж SQLite-лока), без приоритетов.
- Зависимости по данным (carve после facet, questions после facet) держит ПОРЯДОК ЗАПУСКА
  прогонов (по отмашке, последовательно), не мозг.

## Кап по рту — предохранитель (✅ ПОСТРОЕНО, проверено прогоном)

Назначение — НЕ оптимизация, а колпак от «странного поведения»: рот в цикле/шторме не должен осушить пул и заморить остальных.

**Что УЖЕ защищает** (прогон исполнения): per-ключ шаг (`step_for=60·RPM_DIVISOR/RPM`) + per-key cooldown глушат ТЕМП — сломанный рот не пулемётит, задолбанный 429-ключ остывает сам. Глобального аггрегата НЕТ: `GLOBAL_FLOOR` и глоб-abuse-пауза вычищены (канон §5, §2.5 — душили бы 12 проектов как 1). Кап закрывает второе: рот, медленно точащий пул, упрётся в свою дневную долю.

Механика (в `keybroker.py`):

1. Рот присасывается, называя `consumer`.
2. `acquire`: дневной счёт рта (`consumer_usage`) ≥ его кап (`consumer_cap`, нет записи → `DEFAULT_CONSUMER_CAP`)? → отказ `(None,-1)` + лог `cap_block`.
3. На выдаче — `consumer_usage[consumer]++` (та же транзакция, рядом с per-ключ `usage`).
4. Два предела независимы, оба должны пройти: ключ под RPD (per-ключ) И рот под капом (per-рот).

Таблицы + засев:

```sql
consumer_cap(consumer PK, rpd_cap INT, rpm_cap REAL)          -- конфиг; rpm_cap = задел, НЕ enforce
consumer_usage(consumer, pt_day, count, PK(consumer,pt_day))  -- счёт дня, ++ на выдаче
-- DEFAULT_CONSUMER_CAP = 300 (env KB_DEFAULT_CONSUMER_CAP) — незаписанный рот, НИКОГДА не uncapped
```

- `set_cap(consumer, rpd_cap)` — задать кап (числа юзера). `CAPS` + `seed_caps()` засевают значения из таблицы выше; `__main__` сеет при init.
- Проверено: кап 3 → ровно 3 выдачи, дальше отказ; `cap_block` пишется; засеяно 7.

### Логирование аномалий (говно видно в `stats()`)

`request_log.event` пишет не только `grant`/`report`, но и:

- `cap_block` — рот упёрся в свою долю (сигнал: рот тяжёлый/сломан);
- `parse_fail` — пришёл 200, но JSON-мусор (раньше выглядел успехом);
- `429_env` — 429-волна СРЕДЫ (другой ключ словил 429 в окне 30с): ключ НЕ наказан
  (механизм вина-vs-среда, 2026-07-20 — внешний чих больше не прожигает пул в cooldown);
- (глоб-`abuse_pause` вычищена, §канон 2.5 — залп 429 гасится per-key cooldown, не паузой пула).

**Тело не-200 → `error_bodies.log`** (рядом с `keybroker.db`). `call` на ЛЮБОЙ не-200 (400/429/5xx/сеть)
пишет `ts\tstatus\tconsumer\tmodel\tтело[:800]`. Зачем: в `request_log` тела НЕТ, а причина отказа
(RESOURCE_EXHAUSTED? INVALID_ARGUMENT? SAFETY?) — только в теле. **Замер 2026-07-18:** 400 ≈ 429 по числу
(12/12 за сутки), сидят на `consolidate`, НЕ размер (120 меток/90 мух/350 мух-47К-ток → все 200) — тело покажет.

⚠️ **Остаток:** `RPM_DIVISOR=4` (шаг 16с) ещё в коде — под-обслуживает перевод (положено 13/мин). Выпил + темп от нормы рта — отдельный шаг, при проводке translate. Кап это не трогает.

## Инварианты

- Ключи — только в сиське. Рот ключ не получает никогда (ниппель).
- Один сосок. Обойти = взять ключ мимо `call` = запрещено (иначе осьминог обратно).
- `usage` считает расход билдера+ртов. Когда придёт экстрактор — переключить на общую `gemini_quota` (его расход тоже виден). Пока не нужно.
- TPM в сиське НЕ считается: батчи под окном 16.6К (`TPM/RPM`), RPM упирается первым. Счёт токенов понадобится только если батч перевалит за окно.
